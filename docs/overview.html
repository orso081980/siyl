03 Authentication

✓

Secure implementation: HttpOnly cookies + memory tokens

Refresh tokens stored in HttpOnly cookies, access tokens kept in memory only. Protected against XSS attacks.

✓

Current — HttpOnly Cookies + Memory Tokens

  1. 1.Login POST to /api/auth/login → server validates credentials
  2. 2.Symfony sets HttpOnly refresh cookie: Set-Cookie: refresh_token=...; HttpOnly; Secure; SameSite=Lax
  3. 3.Server returns access token in JSON response (short-lived, 15 min)
  4. 4.Frontend stores access token in memory only (not localStorage)
  5. 5.Apollo sends Authorization: Bearer <access_token> + credentials: include
  6. 6.On 401, frontend uses refresh cookie to get new access token

Security: Refresh tokens inaccessible to JS, access tokens evaporate on page reload.

✗

Avoided — JWT in localStorage

  1. 1.Login mutation → server returns { token, role, email }
  2. 2.AuthContext.login() writes to localStorage["mssf_auth"]
  3. 3.Apollo authLink reads localStorage before every GraphQL request
  4. 4.Injects Authorization: Bearer <token> header
  5. 5.Symfony firewall validates JWT (stateless), no session created
  6. 6.errorLink auto-clears storage + redirects to /login on 401

Risk: XSS can steal the token. localStorage is accessible to all JS on the page.

JWT Configuration (Symfony)

Algorithm

RS256 (asymmetric)

Access Token TTL

900 seconds (15 min)

Refresh Token TTL

604800 seconds (7 days)

AppUserProvider resolves the entity (Admin → Professional → User) by email at JWT decode time. Authorization is enforced inside each resolver, not at the firewall level — the /graphql route is PUBLIC_ACCESS.

State & Data Management

▸AuthContext — React Context holding { token, role, email }. Seeded from localStorage on app init.
▸Apollo InMemoryCache — All remote data is cached. Default policy: cache-and-network (shows cached immediately, refetches in background).
▸react-hook-form + Zod — All forms. Validation schemas co-located with form components.
▸No global state library (Redux/Zustand). Apollo handles server state, React Context handles auth.

08 Security Notes

✓ What's in place

  • ✓RS256 asymmetric JWT signing (harder to forge than HS256)
  • ✓Resolver-level authorization (each resolver checks current user role)
  • ✓Ownership checks (users can only access their own data)
  • ✓Password hashing (Symfony's password hasher)
  • ✓CORS configured via nelmio/cors-bundle
  • ✓Auto-logout on 401/403 in Apollo errorLink
  • ✓Message content limited to 5000 chars
  • ✓ILIKE search (not raw injection) via Doctrine native queries

⚠ Areas to improve

  • !JWT stored in localStorage — XSS-vulnerable. Prefer HttpOnly cookie.
  • !No token refresh mechanism — tokens expire after 1h with no silent refresh.
  • !No rate limiting on login/register mutations.
  • !No GraphQL depth/complexity limiting (DoS via deeply nested queries).
  • !No file upload validation (avatar URLs are plain strings — no server-side check).
  • !No CSP headers configured on Symfony responses.
